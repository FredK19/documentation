WikiText Guide

Concept Overview

WikiText is the easiest way to maintain formatting and consistency across a vast number of authors and documents. Using WikiText instead of HTML insures that all text will seamlessly adhere to formatting requirements for marketplaces, syndication feeds, and templates thereby ensuring the most professional presentation. When compared to HTML you'll find WikiText is also considerably easier to learn, easier to read/edit, and guaranteed to avoid all the known issues surrounding embedded HTML tags.

Possibly one of the most impressive properties of wikitext is that each template (collectively site wrapper, auction wizard, and/or page layout) applies it's own unique style to wikitext. Using WikiText guarantees your text will always be in the proper color, size, and font that the template author intended.

Later on if you decide to change templates then the text automatically changes to match the styles of the new template!

WikiText is supported in every Text Field by Zoovy. It can be used in product descriptions, shipping and return policies, any other place you need to format large blocks of text!
If you choose to embed HTML tags instead of using wikitext you will spend more time maintaining your site and may encounter errors when working with marketplaces.

Basic Syntax Overview

Place each wiki command, one per line, the leading character must always be in column 1. You can use WikiText commands

Titles

WikiText specifies three types of title, examples are as follows:
= Section =
== Sub Section ==
=== Sub Sub Section ===
Your H1, H2 and H3 text colors should be either black, or a color that matches your site theme.
= Section (H1)= == Sub Section (H2) == === Sub Sub Section (H3) === Just make sure the first characters and last characters on any line are =, ==, === and you'll get a title
A section title will typically skip down a line, and may have some sort of formatting above, below or adajcent (such as a graphic, or underline). A subsection title is the most commonly used and is used to break text and identify key topics of text. A sub sub section is really a minor title, typically sub sub sections will not include any line breaks and will include minimal formatting.

New Paragraphs

To begin a new paragraph either use a = title =, or == sub-title ==, or if you don't want a header then simply insert two blank lines.

Bullets

Bullets are perhaps the most important feature of wikitext. To create a bullet list simply start each line with an asterisk (*), and enter one bullet per line. Here is an example: 
* Bullet item 1
* Bullet item 2
* Bullet item 3
Amazon estimates that items which contain bullet lists are 65% more likely to sell. Incredible!
Some marketplaces (specifically Amazon) will extract the first wiki bullet list found in your product description and use that for the products bullet list. Amazon formats bullet lists differently, and displayed to users before the product description. Always make sure the first bullet list in your product description is your primary bullet list.
It is also very common (recommended) to create a bullet list with a subsection header e.g. 
== Features ==
* Feature 1
* Feature 2

Advanced Syntax

Section Dividers

Insert four or more dashes (e.g. ----) on it's own line to insert a line or section break. Usually this will be translated as a
in HTML, and is commonly used to signify a totally new unrelated topic of text that follows.

Tables

Creating a table in your text is easy, just use the following syntax: 
|=Heading A|=Heading B|=Heading C
|1a|1b|1c
|2a|2b|2c
The lines with |= are considered header cells, they will have special formatting to them, if you wanted to create a two column vertical table which highlighted the values rather than the headers, you could do something like this: 
|Header 1|=Value 1
|Header 2|=Value 2
|Header 3|=Value 3
|Header 4|=Value 4

Linking with Wiki

There are several different ways to link with wiki: 
[[keyword]]  -  will link the word 'keyword' to a search results set for 'keyword'
[[Linked phrase]:search=Cable]  - will link the phrase to a search results set for 'cable'
[[visit a domain]:url=http://www.domain.com]  - will link phrase 'visit a domain' to www.domain.com
[[open a window]:popup=http://someurl.com] - will link 'open a window' to any location AND open in a new window (not a popup)
[[emailus]:url=mailto:name@email.com] - will link 'emailus' with a mailto tag that sends to name@email.com
[[category text]:category=your.safe.path] - will link 'category text' the category page .your.safe.path
[[product text]:product=yourproductid] - will link 'product text' the product page for the yourproductid product.
[[phrase]:customer=login] or [[phrase]:customer=signup]
Some common values for the customer= would be: 
[[Login]:customer=login]
[[Logout]:customer=logout]
[[Recover Password]:customer=login]
[[My Account]:customer=order]
[[Quick Order]:customer=wholesale/order]
[[Change Password]:customer=password]
[[Email Preferences]:customer=newsletter/config]
[[Change Password]:customer=password]
[[Order Status]:customer=order/status]
[[Create Account]:customer=order/signup] (feature requires wholesale bundle Doc 50512)
At the end of the url/search/product/etc, you can set &:CLASSNAME to set a specific class name on a link. This will allow developers to override the link styles and/or behaviors based on the class name. Example: [[phrase]:customer=login:&ztitle] would add the class ztitle to the link (which is a default zoovy class in all/most themes).

Escaping special characters in wiki links

The wiki linking convention requires special characters to be encoded. The equal sign, for instance, is part of the wiki syntax and if one is present in the URL, then it will be interpreted by the wiki engine and the wiki link will not work.

Disabling WikiText

To disable wikitext command processing in the element of the template set WIKI="0"

Looking Ahead: WikiText relevance in the Future

The following section describes how using WikiText can help future proof your business! Not only is it easy, but it's also important to make sure your product data can be used in places other than your website and auction templates.

WikiText + Ajax (Web 2.0)

New website programming techniques (specifically one called "AJAX") will alter the face of every website and make heavy use of DHTML and Javascript. Both DHTML and Javascript have compatiblilty issues when outputting HTML (especially broken/non structured XHTML) which can cause browser memory leaks, crashes, or in a best case scenario heavily mangled text. Site owners who intend to deploy Ajax based websites which offer a substantially improved buyer experience will likely need to remove any embedded HTML for those sites to work properly.

Amazon (the unofficial "King" of personalization) has been working on an AJAX interface for several months already. Although it has not been released, most experts agree that AJAX interfaces can provide such substantial user interface improvements that customers will be much more likely to make a purchases exclusively from Ajax based websites. Higher Volume = Lower Prices = More Competition. In the real world having a non-ajax website would be the equivalent to having a store on "the bad side of town".

NOTE: Zoovy is already working on AJAX templates, the rendering system upgrades in Sept. 2005 were intended to bring various components of the system into compliance so we could start implementing this. Expect to see the first series of AJAX based websites around Feb. 2006.

WikiText + Google

Google introduced AJAX programming to the world with innovative services such as Google Suggest and Google Earth - so it's no surprise they see the writing on the wall and how AJAX will affect their business.

With the advent of AJAX based websites, websites will become much harder to spider. (NOTE: Spidering is how google builds it's search engine database, without an alternative to spidering Google would be effectively out of business). Google recognizes this emerging trend and has already introduced Googlebase and Froogle which allow website owners to feed content directly into Googles database. GoogleBase will eventually be the preferred method, for Google to receive structured content, and it's reasonable to assume that they will offer some limited markup abilities, but NOT allow a full HTML markup due to security issues with cross site scripting. (Google will never allow HTML in their XML/RSS/ATOM feeds since those feed formats discourage/disallow the use of embedded HTML. WikiText will likely be the markup they choose, or at a minimum wikitext can easily be translated into any format Google requests.

The Mobile Web

In 2006 we are going to see some rapid deployment of mobile devices, and more specifically Internet equipped mobile devices, as well as the probable elimination of several regulatory barriers, which will cause all site owners to evaluate making their websites available on the "Mobile Web". WikiText can be translated as either HTML or WML (Wireless Markup Language).
It's also worth mentioning that with Googles penchant for mobile devices and the wireless Internet, and they're resulting horde of structured data from Googlebase, they're accurate and detailed knowledge of city maps and locales from Google Earth, and they're fetish for PhD's with ample free time we'll be seeing absolutely some incredible innovative wireless applications in 2006/2007. Quite honestly, by the end of 2006 the entire world will have changed.

You'll start to see the applications towards the end of 2006, in anticipation of the big changes in 2007. First all non-digital television broadcasts will go dark in July 2007. The important side note is that the FCC has already said they plan to resell/relicense that radio spectrum. This will make a variety of high speed, low power, wide area Internet services available and incredibly affordable. The UHF spectrum is widely regarded as the cherry spot for consumer wireless, and is why many countries outside the US have much better mobile Internet options than we do.

NOTE: Zoovy is already working on providing WML based websites to mobile devices.
In conclusion, WikiText is a way of future-proofing your business and ensuring that your data will be able to participate in the upcoming technology revolution. While WikiText by itself doesn't guarantee success, any business owner knows that particiption is requisite for generating sales.

Understanding Wiki Translation Engine - a designers point of view

The beautiful part about formatting a site with Wiki is the flexibility it gives the designer of the site to maintain complete control over the aesthetic properties of the site by wrestling that control away from the person writing the content.
Lets use a simple example to illustrate how the Wiki translation engine works, in this example lets say a user creates a simple bulleted list in wiki like such:
* Item 1 * Item 2 * Item 3
The Zoovy Wiki engine will translate that syntax into an intermediate format that would appear as such:
%list% %listitem%Item 1%/listitem% %listitem%Item 2%/listitem% %listitem%Item 3%/listitem% %/list%
The designer can then specify SUBs in the wrapper of the site which override the %list%, %listitem%, %/listitem%, and %/list% tags during site rendering to produce ANY of the following results: Simple bulleted list, with CSS class:
  • Item 1
  • Item 2
  • Item 3
Or even create a complex table with graphical bullets:
Item 1
Item 2
Item 3
The content author does not need to be concerned with the display of the output, that is up to the designer.
WIKI syntax also means that in the future sites such as Amazon, Google, etc. will be able to use WIKI Markup to ensure that product descriptions (regardless of author or origin) appear consistently throughout their marketplace and can avoid the "eBay Effect" where each listing has it's completely unique presentation thereby hurting the overall shopping experience of the customer.
