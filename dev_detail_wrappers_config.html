Theme Configuration

CSS

The CSS attribute is a node for your CONFIG element. This css will be output in your HEAD element when the theme renders. You can have your own styles in the theme itself (or an external style sheet) as needed, but you should not add your own styles or css attributes to the zoovy styles.

Where are these used? There is no specific set of rules but below are a list of guidelines. The styles, in most cases, are not used within the theme, but in the layouts themselves. This is done so that if the theme is changed, the layouts will adopt the color properties of the new theme and have a pretty good chance of looking good. Some styles have specific uses as well.
ztable_... - these are used in 'lists', such as product lists, category lists and in the shopping cart elements. The row0 and row1 are the alternating colors that get used if the list spec supports that feature. Typically, row0 would get used as the first row, then alternate from 1 to 0 for all the remaining rows. ztable_row_small would get used when smaller text is needed and ztable_row_title would get used for titles.
zform... - these styles are used in the various forms output by Zoovy. Going forward, these styles will be applied to all forms. Which class is assigned to which form element is pretty straightforward. In some cases, additional classes or ID's are present so that specific buttons can have a separate style associated.

ztab... - Used in cases where 'tabs' are present, such as in product layouts and, someday, perhaps one page checkout. tab0 is the 'off' class and tab1 is the 'on' class. ztabbody is the class that would be on the content of the tabbed data.
ztxt - this is the default class that will be applied to body copy. All TEXT elements are output with a font tag with ztxt class set (unless FONT="0")
ztitle & ztitle2 - often used for styling on headings and sometimes for wiki.
zwarn - used for error handling.

Wrapper 'hinting' for layouts

Currently, there is one supported hint to let a layout know something about the wrapper around it. This was introduced for One Page Checkout (1PC) but will likely grow beyond it.
set IHAVEJQUERYUICSS="1" on the config element of the wrapper. This lets the checkout layouts know that jquery ui css is loaded and to not load the layout default (which is most likely cupertino).

Developer (site configurator) Overrides:
    - dev.no_subcategories : set to 1 to disable subcategories inside layouts.
    - dev.no_continue : set to 1 to disable continue button on cart AND cancel button on product page.
    - dev.softcart : set to 1 to enable softcart
    - dev.softcart_referrers : set to 1 to enable softcart referrer tracking
    - dev.killframes : set to 1 to put target="_top" on category links (if category element is properly configured)
    - dev.ssl_only : force entire store to SSL (not recommended)
    - dev.search_bounce : do not bounce to search.cgi page when no results are returned from query on results.cgi
    - dev.customer_management : set to private to require login
    - dev.login_form_suppress : set to 1 to not display the login form on the login page (allows you to customize the appearance/placement of the form in the layout).
Misc:
    - site.css : set to 1 or 0 and it will disable/enable autogenerated font tags site wide. (outside of specs)
    - flow.product : set to document id for force that document as a layout on product pages
    - flow.category : set to document id for force that document as a layout on category pages
    - flow.homepage : set to document id for force that document as a layout on homepage. you can pass flow.search, flow.returns or any other zoovy page as well. flow.checkout is NOT SUPPORTED.
    - defaultflow.c : default all category pages to a specific layout.
    - defaultflow.p : default all product pages to a specific layout.
404 page - this page does NOT do any cart updating, therefore you can NOT do any ajax or element updates that save to the CART: namespace. Post the form to the cart page.
Consistency is key to a website. For products, we recommend you force the same layout in your wrapper so that all of your product detail pages look the same.
For category pages, use the defaultflow overload, but never force them. Category pages need flexibility because they're used not just for pages that feature subcategories and/or product, but for any 'other' type of page that is needed.

Sitebuttons

There are 9 sitebuttons that are used in your Zoovy store (listed below). Your buttons can be either JPG, PNG, GIF or a combination thereof. Buttons should be named as domaincom_ + the name of the button (where domaincom is the domain of the store they're going to be used on. ex: autopartscom_). For instance, domaincom_add_to_cart.gif. The text on the button itself can read differently (ex: forward could say 'next') but the buttons MUST be named as follows (case sensitive):
add_to_cart add_to_site (used for JEDI) back cancel checkout continue_shopping empty_cart forward update_cart

Once you have the buttons created, upload them to your public files directory (go to setup, then 'public files directory' under the media heading). Once uploaded, go in to your custom theme and remove any reference to existing sitebuttons from the config element. Sitebutton Strategies

Button Text

The buttons can say whatever you want, however it should be indicative of what the button does. Changing the text will NOT change the action of the button. The button 'empty cart' should probably always read 'empty cart' or 'empty'. However, the 'continue shopping' button could read 'shop more'. The forward button can, and probably should, read 'next' as that is more familiar. It is also common, for some industries, for the 'add to cart' button to read 'add to shopping bag'.
It is important that you are consistent in your naming. So if the add button reads 'add to shopping bag', then all other references to the cart page should be changed to 'shopping bag' so that you are consistent. This will help your customers identify what the 'bag' is and it will appear more professional.

Button Color

In most cases, the sitebuttons appear in groups of two or four/five. The groups are as follows:
Product detail page: add to cart and cancel
Checkout: forward and back
Cart: empty cart, update cart, continue shopping, checkout and add to store.
Each of these sets of buttons will have one button that, most likely, you want clicked more than the rest. For instance, on the cart page you want 'checkout' to be clicked. So it stands to reason that the checkout button should stand out more than the other buttons. The same is true for the add to cart and next buttons.

Dimensions

Depending on the layout you have selected for the various pages, the sitebuttons will appear in a horizontal row or on top of each other. Since this is the case, the best aesthetic for these will typically have all the buttons in a given group share the same dimensions. A row of buttons looks funny if one button in the row is a different height. Vertical buttons don't look as nice when they don't share the same width.
